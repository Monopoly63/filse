الفصل الأول · الأساسيات

ما هي الشجرة الحمراء-السوداء؟

A Red-Black Tree is a self-balancing binary search tree where each node carries an extra bit of color — Red or Black.

Invented by Rudolf Bayer in 1972 (as "symmetric binary B-trees"), it was later renamed. It's the underlying structure in many real systems: Java TreeMap, C++ std::map, Linux kernel (CFS scheduler), and MySQL indexes.

The color property is not just cosmetic. It encodes a set of rules that guarantee the tree remains approximately balanced at all times — keeping every operation at O(log n) without the expensive full rebalancing of AVL trees.

⚡ Core Idea

Each node is either RED or BLACK.

By enforcing 5 simple rules, the tree guarantees:

  • Longest path ≤ 2× shortest path
  • Height ≤ 2 log₂(n+1)
  • Every operation stays O(log n)

Node Structure

Every node in a Red-Black Tree stores:

struct RBNode { int key; // value stored in the node Color color; // RED or BLACK RBNode *left; // left child RBNode *right; // right child RBNode *parent; // parent pointer (needed for fixup) };
RED node — allows violation of BST height rules temporarily
BLACK node — enforces black-height constraints

The parent pointer is essential — insertion and deletion fixup algorithms must walk up the tree toward the root.

Chapter 02 · The Rules

Five Properties (Red-Black Rules)

Every valid Red-Black Tree must satisfy exactly these five properties:

  • 1
    Every node is either Red or Black.
  • 2
    The root is always Black. (This prevents root-based violations of property 4.)
  • 3
    Every leaf (NIL) is considered Black. We model this as a sentinel node in practice.
  • 4
    Red nodes cannot have Red children. No two consecutive Red nodes in any path.
  • 5
    Every path from a node to its descendant leaves has the same number of Black nodes (Black-Height).

💡 Why "5 Properties" Why not a height constraint?

Properties 1–5 imply that the longest path from root to any leaf is at most 2× the shortest path. This is proven from the black-height rule: any path with the minimum black nodes can have at most twice as many red interleaved nodes. This is why insertion/deletion stay O(log n) — the tree is never allowed to become a linked list.

Chapter 04 · Operations

Insertion (Insert-Fixup)

Insertion is a two-phase process: standard BST insert in O(log n), then fix any violations.

1. Standard BST Insert
Insert the new node colored RED using normal BST rules. The RED color signals that we might have violated property 4 (no consecutive reds), and the fixup routine must handle it.
2. Run Insert-Fixup
Walk up the tree from the new node. If its parent is also Red, determine the uncle's color and apply one of the 3 cases below. Stop when the root is reached.
3. Fix the Root
After fixup, always color the root Black. This enforces property 2.

Three Cases

CASE A · UNCLE IS RED

Recolor (Zig-Zig or Zig-Zag)

If the uncle is Red: simply recolor the parent and uncle as Black, recolor the grandparent as red, and continue fixing up from grandparent. No rotations needed.

if (uncle.color == RED) {
  parent.color = BLACK;
  uncle.color = BLACK;
  grandp.color = RED;
  node = grandp; // continue up
}
CASE B · LINE (Left-Left)

Right Rotation + Recolor

If the uncle is Black and the node + parent are in a straight line (node is left-child of left-child OR right-child of right-child): do a single rotation in the opposite direction, then swap colors.

// Left-Left case:
parent.color = BLACK;
grandp.color = RED;
right_rotate(grandp);
CASE C · TRIANGLE (Right-Left)

Left Rotation + Right Rotation

If the uncle is Black and node + parent form a triangle (node is right-child of left-child OR left-child of right-child): rotate on the parent first to convert into Case B, then handle Case B.

// Right-Left case:
node = parent; // start from parent
left_rotate(node); // now becomes Left-Left
// fall into Case B...

Chapter 05 · Advanced Operation

Deletion (Delete-Fixup)

Deletion is the hardest operation. When you remove a Black node, you either reduce the black-height of paths going through that subtree — violating property 5. Fixup must restore it.

🔴 If you delete a RED node

The tree stays balanced. No fixup needed. Simply splice it out like a normal BST node.

⚫ If you delete a BLACK node

A double black phenomenon appears. We must redistribute an extra Black from a sibling through rotations + recoloring. This triggers the famous 6 deletion cases.

Deletion Cases (6 Patterns)

Case D1: Sibling is Red
Recolor parent + sibling, rotate at parent. The new sibling (former parent) is now Black — drops into D2, D3, or D4.
Case D2: Sibling is Black, both nephews are Black
Pull Black from sibling onto parent (color sibling Red, parent Black). The double-Black moves up to parent.
Case D3: Sibling Black, far nephew Black, near nephew Red
Rotate at sibling to convert to Case D4.
Case D4: Sibling Black, far nephew Red
Rotate at parent, recolor parent & sibling with the near nephew Black. This absorbs the double-Black.

Chapter 06 · Building Blocks

Left & Right Rotations

Rotations are the atomic structural operations that preserve the BST ordering while changing the tree's shape. Together with recoloring, they fix violations.

Right Rotation (on X)
↻

Used when heavy-left subtree pulls tree left

X = root
Y = X.left
X.left = Y.right
Y.right = X
return Y as new root
Left Rotation (on X)
↺

Used when heavy-right subtree pulls tree right

X = root
Y = X.right
X.right = Y.left
Y.left = X
return Y as new root

🎯 Key Property of Rotations

Rotations run in O(1) — just pointer reassignments. They do not touch the rest of the tree or change any keys. BST ordering is preserved because the middle node is simply elevated, and its former single child becomes the dangling subtree of its former parent.

Chapter 07 · Which One?

Red-Black Tree vs AVL Tree

Both are self-balancing BSTs. They differ in when and how much they rebalance.

RED-BLACK TREE
  • Max height: ≈ 2 log₂ n
  • Worst insert: O(log n)
  • At most 2 rotations per insert
  • Cheaper rebalancing
  • Better for write-heavy workloads
AVL TREE
  • Max height: ≈ 1.44 log₂ n
  • Worst insert: O(log n)
  • Up to O(log n) rotations (≈1.44 average)
  • Faster lookups (stricter balance)
  • Better for read-heavy workloads

C++ std::map uses Red-Black. C++ std::set also uses Red-Black. Java TreeMap follows suit. This is the industry standard for ordered maps.

Chapter 08 · Performance

Time & Space Complexity

Because the tree stays balanced within a constant factor of the optimal height, every operation runs in logarithmic time.

Operation Best Average Worst Notes
Search O(1) O(log n) O(log n) Traverses height path
Insert O(log n) O(log n) O(log n) + ≤2 rotations + O(1) fixup
Delete O(log n) O(log n) O(log n) + ≤3 rotations + O(log n) fixup
Find Min/Max O(log n) O(log n) O(log n) Follow leftmost/rightmost path
Traverse (all nodes) O(n) O(n) O(n) Inorder = sorted output

Space complexity is O(n) — each node stores 5 fields. No extra heap beyond the node storage. Call-stack for recursive operations is ≤ tree height = O(log n).

Chapter 10 · Implementation

Core Pseudocode

Left Rotation

LEFT-ROTATE(T, x)
  y ← x.right // set y
  x.right ← y.left
  if y.left ≠ NIL
    y.left.parent ← x
  y.parent ← x.parent
  if x.parent = T.nil
    T.root ← y
  else if x = x.parent.left
    x.parent.left ← y
  else x.parent.right ← y
  y.left ← x
  x.parent ← y

Right Rotation

RIGHT-ROTATE(T, x)
  y ← x.left // set y
  x.left ← y.right
  if y.right ≠ NIL
    y.right.parent ← x
  y.parent ← x.parent
  if x.parent = T.nil
    T.root ← y
  else if x = x.parent.right
    x.parent.right ← y
  else x.parent.left ← y
  y.right ← x
  x.parent ← y

RB-INSERT-FIXUP (Outline)

while z.parent.color == RED
if z.parent == z.parent.parent.left
  if uncle.color == RED → Case A: recolor & continue up
  else if z == z.parent.right → Case C: rotate left at parent
  else → Case B: rotate right at grandparent + recolor
mirror same logic for right branch
T.root.color = BLACK

Chapter 11 · Real-World Usage

Where are Red-Black Trees Used?

🗺️

std::map / std::set (C++)

The GNU C++ Standard Library implements ordered maps with Red-Black trees.

☕

Java TreeMap / TreeSet

Java's TreeMap and TreeSet use Red-Black trees internally for guaranteed O(log n) ordered operations.

🐧

Linux Kernel

The Completely Fair Scheduler (CFS) and virtual memory subsystem use Red-Black trees extensively.

💾

Database Indexes

MySQL and PostgreSQL use B+ trees for disk-based indexes, but the in-memory components often use Red-Black variants.

⚡

Multithreading / epoll

Red-Black trees are used to manage timers and epoll red-black trees in the Linux kernel.

📦

Java HashMap / ConcurrentSkipList

Java 8+ HashMap switches to a balanced tree when a bucket overflows — a Red-Black tree variant.
